Introduction

What is RAG and Why Self-Corrective?

RAG (Retrieval-Augmented Generation) is a pattern where an LLM answers questions by first retrieving relevant documents from a knowledge base, then generating an answer grounded in those documents.

Basic RAG has a simple flow: retrieve → generate. But this breaks in three ways:

Problem 1: The user's question may not match documents well. A vague question like "what's that tall thing in Paris?" won't find "Eiffel Tower" in vector search.
Problem 2: Retrieved documents may not actually be relevant. Vector similarity returns "similar" text, not necessarily "useful" text.
Problem 3: The LLM may hallucinate — making up facts that aren't in the retrieved documents.

This pipeline fixes all three with query rewriting, relevance grading, web search fallback, and hallucination checking with retry.

Step 1

Set Up the Models

We need two AI models: one to convert text into vectors (embeddings), and one to generate text (LLM).

embeddings.py — Text → Vectors

from langchain_google_genai import GoogleGenerativeAIEmbeddings

def get_embedding_model():
    """Converts text into 3072-dimensional vectors."""
    return GoogleGenerativeAIEmbeddings(
        model="models/gemini-embedding-001",
        google_api_key=GOOGLE_API_KEY,
    )

llm.py — The Brain

from langchain_openai import ChatOpenAI

def get_llm(temperature=0):
    """OpenAI GPT-4o-mini — generates answers."""
    return ChatOpenAI(
        model="gpt-4o-mini",
        temperature=temperature,
    )
Why two models? Embeddings are cheap and fast (just converting text to numbers). The LLM is smarter but slower — we only use it for tasks that need reasoning.
Step 2

Store Documents in Qdrant

Before we can search, we need to store documents. Each document gets chunked into small pieces, embedded into vectors, and stored in Qdrant Cloud.

chunker.py — Split Text into Chunks

def chunk_text(text, chunk_size=500, overlap=50):
    """Split long text into overlapping pieces.
       Overlap ensures we don't lose context at boundaries."""
    chunks = []
    start = 0
    while start < len(text):
        chunk = text[start : start + chunk_size].strip()
        if chunk:
            chunks.append(chunk)
        start += chunk_size - overlap
    return chunks

database.py — Store and Search

def add_documents(texts):
    """Chunk → embed → store in Qdrant."""
    store = get_vector_store()
    all_chunks = []
    for text in texts:
        all_chunks.extend(chunk_text(text))
    store.add_texts(all_chunks)
    return len(all_chunks)

def search_with_scores(query, top_k=4):
    """Find similar docs with similarity scores."""
    store = get_vector_store()
    results = store.similarity_search_with_score(query, k=top_k)
    return [(doc.page_content, score) for doc, score in results]
Step 3

Define the Pipeline State

LangGraph needs a typed state — a structured object that flows through every node. Think of it as a form that gets filled in step by step.

state.py — The Data That Flows

class RAGState(TypedDict):
    question: str                     # what the user asked
    rewritten_query: str              # improved query for search
    documents: list[tuple[str, float]] # retrieved (text, score) pairs
    relevant_docs: list[str]          # docs that passed grading
    web_results: str                  # web search fallback results
    answer: str                       # the generated answer
    retry_count: int                  # generation attempts
    is_grounded: bool                 # hallucination check result
    steps: Annotated[list[str], merge] # trace log
The steps field uses an Annotated reducer — this means each node appends to the list instead of replacing it. Every node adds a log message, building a complete trace of what happened.
Step 4

Build the 6 Node Functions

Each node is a plain Python function: it receives the current state and returns the fields it wants to update. All 6 nodes live in nodes.py.

NodeWhat it does
rewrite_queryLLM rewrites the user's question for better vector search
retrieveSearches Qdrant for the 4 most similar document chunks
grade_documentsLLM grades each doc: "Is this actually relevant?" (yes/no)
websearch_fallbackIf no relevant docs, search the web via DuckDuckGo
generateLLM generates the final answer from the best available context
check_hallucinationLLM verifies: "Is this answer supported by the context?"

There are also two routing functions that decide which path to take:

def route_after_grading(state):
    """Relevant docs found? → generate. None found? → web search."""
    if state["relevant_docs"]:
        return "generate"
    return "websearch_fallback"

def route_after_check(state):
    """Grounded? → done. Hallucinated + retries left? → retry."""
    if state.get("is_grounded", True):
        return "accept"
    if state.get("retry_count", 0) < 2:
        return "retry"
    return "accept"
Step 5 — The Core

Wire the Graph

This is where LangGraph shines. We connect nodes with edges, conditional branches, and a retry loop.

Node (function)
Edge (always)
Branch (conditional)
Loop (retry)
graph.py
graph = StateGraph(RAGState)

# 6 Nodes — the steps
graph.add_node("rewrite_query", rewrite_query)
graph.add_node("retrieve", retrieve)
graph.add_node("grade_documents", grade_documents)
graph.add_node("websearch_fallback", websearch_fallback)
graph.add_node("generate", generate)
graph.add_node("check_hallucination", check_hallucination)

# Edges — fixed connections (green solid arrows)
graph.add_edge(START, "rewrite_query")
graph.add_edge("rewrite_query", "retrieve")
graph.add_edge("retrieve", "grade_documents")
graph.add_edge("websearch_fallback", "generate")
graph.add_edge("generate", "check_hallucination")

# Branch — conditional routing (amber dashed arrows)
graph.add_conditional_edges(
    "grade_documents", route_after_grading,
    {"generate": "generate", "websearch_fallback": "websearch_fallback"}
)

# Loop — retry cycle (red dashed arrows)
graph.add_conditional_edges(
    "check_hallucination", route_after_check,
    {"accept": END, "retry": "generate"}
)

Key Insight

Why LangGraph?

A simple chain (retrieve → generate) can't do what this pipeline does. Here's why:

Conditional Branch — After grading, the pipeline picks one of two paths. A chain can't branch.
Retry Loop — If the answer hallucinates, the pipeline goes back to generate and tries again. A chain can't loop.

LangGraph gives you a graph instead of a chain. Graphs can branch and loop — that's the fundamental difference.
